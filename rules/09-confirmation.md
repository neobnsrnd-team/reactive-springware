# 09-confirmation.md — 개발자 확인 원칙

## 목적

Claude가 코드 생성 중 판단할 수 없는 구간에서 임의로 진행하지 않고,
반드시 개발자에게 확인을 받은 후 진행하도록 강제한다.

---

# 기본 원칙

확인이 필요한 상황에서는 **절대 임의로 가정하고 진행하지 않는다.**
불확실한 내용을 추측하여 생성하면 잘못된 코드를 수정하는 비용이 처음부터 확인하는 비용보다 크다.

확인이 필요한 경우 다음 형식으로 질문한다.

```
확인이 필요합니다.

[확인 항목]
- 질문 1
- 질문 2

확인 후 진행하겠습니다.
```

---

# 1. Figma 해석 불명확

## 컴포넌트 매핑 불명확

Figma 컴포넌트가 `component-map.md`에 없는 경우, 임의로 컴포넌트를 만들지 않는다.

```
확인이 필요합니다.

[확인 항목]
- Figma의 "{컴포넌트명}" 컴포넌트가 component-map.md에 정의되어 있지 않습니다.
- 가장 유사한 컴포넌트는 "{유사 컴포넌트}"입니다. 이것으로 대체할까요?
- 아니면 새로운 컴포넌트를 추가해야 할까요?
```

## 엔티티명 불명확

Figma 디자인에서 어떤 도메인/엔티티인지 판단할 수 없어 네이밍을 결정할 수 없는 경우.

```
확인이 필요합니다.

[확인 항목]
- 이 화면의 엔티티명(도메인명)을 확인해주세요.
- 엔티티명에 따라 파일명, Hook명, Repository명이 결정됩니다.
  예: "User" → UserListPage, useUserList, userRepository
```

# 2. API / 데이터 구조 불명확

## API 엔드포인트 불명확

어떤 API를 호출해야 할지 알 수 없는 경우, Repository를 임의로 생성하지 않는다.

```
확인이 필요합니다.

[확인 항목]
- 이 화면에서 호출할 API 엔드포인트를 확인해주세요.
  예: GET /api/users, POST /api/users
- 엔드포인트를 알아야 Repository를 정확하게 생성할 수 있습니다.
```

## 서버 응답 타입 불명확

서버 응답 구조를 알 수 없거나, nested 데이터가 존재하여 Types를 정의할 수 없는 경우.

```
확인이 필요합니다.

[확인 항목]
- API 응답 데이터의 구조(필드명, 타입)를 확인해주세요.
- 응답 구조를 알아야 타입 정의와 데이터 가공 로직을 정확하게 생성할 수 있습니다.
```

nested 데이터가 있는 경우 특히 주의한다.

```
// 예: 응답 구조가 아래처럼 중첩된 경우 매핑 방식을 반드시 확인
{
  data: {
    items: [],
    meta: { total: 0, page: 1 }
  }
}
```

## Pagination 구조 불명확

API의 pagination 방식이 불명확하여 Repository의 요청 파라미터와 응답 처리를 결정할 수 없는 경우.

```
확인이 필요합니다.

[확인 항목]
- 이 API의 pagination 방식을 확인해주세요.
  1. page / size 파라미터 방식  예: GET /users?page=1&size=10
  2. offset / limit 파라미터 방식  예: GET /users?offset=0&limit=10
  3. cursor 방식
- pagination 응답 구조도 확인해주세요.
  예: { items: [], total: 0 } 또는 { data: [], meta: { total, page } }
```

## Filter / Search 파라미터 불명확

검색·필터 파라미터 구조를 알 수 없어 Repository 요청 구조를 결정할 수 없는 경우.

```
확인이 필요합니다.

[확인 항목]
- 검색/필터 파라미터 구조를 확인해주세요.
  예: GET /users?keyword=홍길동&status=active&page=1
- 어떤 필터 조건이 지원되는지 확인해주세요.
```

## 필드 매핑 불명확

서버 필드명과 클라이언트 모델 필드명이 달라 매핑 방법을 결정할 수 없는 경우.

```
확인이 필요합니다.

[확인 항목]
- 서버 응답 필드명을 클라이언트 모델 필드명으로 어떻게 매핑할지 확인해주세요.
  예: user_id → id, user_name → name
```

# 3. 화면 타입 해석 불명확

## Table vs Card vs Form 판단 불명확

Figma 디자인의 반복 UI 구조가 Table인지 Card 목록인지 판단할 수 없는 경우,
또는 Form 구조가 단순 입력인지 복잡한 구조인지 판단할 수 없는 경우.

```
확인이 필요합니다.

[확인 항목]
- 이 반복 UI의 표시 방식을 확인해주세요.
  1. Table — 행/열 구조로 데이터를 나열하는 경우
  2. Card — 카드 형태로 데이터를 나열하는 경우
- 선택에 따라 사용하는 컴포넌트가 달라집니다.
  Table → Card 반복 렌더링 또는 TransactionList
  Card → Card 반복 렌더링
```

Form 구조가 애매한 경우:

```
확인이 필요합니다.

[확인 항목]
- 이 Form의 구조를 확인해주세요.
  1. 단순 입력 폼 (필드 나열)
  2. 섹션 구분이 있는 복합 폼
  3. 단계별 입력 폼 (step/wizard)
```

# 4. Form 제출 로직 불명확

## Create vs Update 판단 불명확

같은 Form 컴포넌트가 등록(create)과 수정(update) 양쪽에 쓰이는지, 별도 페이지인지 판단할 수 없는 경우.

```
확인이 필요합니다.

[확인 항목]
- 이 Form이 다음 중 어떤 용도인지 확인해주세요.
  1. 등록 전용 (POST /api/users)
  2. 수정 전용 (PUT /api/users/:id)
  3. 등록·수정 공용 (id 유무로 분기)
```

## Validation 방식 불명확

필드 유효성 검증 방식을 결정할 수 없는 경우.

```
확인이 필요합니다.

[확인 항목]
- 폼 유효성 검증 방식을 확인해주세요.
  1. 필수값(required)만 처리
  2. 형식 검증 포함 (이메일, 전화번호 등)
  3. 서버 측 검증 결과를 필드에 표시 필요
- 사용 중인 validation 라이브러리가 있다면 알려주세요.
  예: react-hook-form, zod, yup
```

# 5. 기존 코드와 충돌

## 동일 파일명 충돌

생성하려는 파일이 이미 존재하는 경우, 덮어쓰지 않는다.

```
확인이 필요합니다.

[확인 항목]
- "{파일명}" 파일이 이미 존재합니다.
  1. 기존 파일을 덮어쓸까요?
  2. 기존 파일에 내용을 추가할까요?
  3. 새로운 이름으로 생성할까요?
```

## 라우터 URL 충돌

등록하려는 URL이 기존 라우터 경로와 겹치는 경우.

```
확인이 필요합니다.

[확인 항목]
- "{URL}" 경로가 기존 라우터에 이미 등록되어 있습니다.
  1. 기존 라우터를 수정할까요?
  2. 새로운 URL 경로를 사용할까요?
```

# 6. 비즈니스 로직 판단 필요

## 단순 CRUD 이상의 복잡한 로직

디자인에서 특정 비즈니스 규칙이 존재하는 것은 보이지만 로직을 추론할 수 없는 경우.

```
확인이 필요합니다.

[확인 항목]
- "{기능명}" 기능의 처리 로직을 확인해주세요.
  예: 상태 변경 조건, 유효성 검증 규칙, 계산 로직 등
- 로직이 확인되어야 Hook과 Repository를 정확하게 생성할 수 있습니다.
```

## 권한/인증 처리 불명확

이 페이지에 접근 권한 제어가 필요한지 판단할 수 없는 경우.

```
확인이 필요합니다.

[확인 항목]
- 이 페이지에 접근 권한 제어(인증, 역할 제한 등)가 필요한지 확인해주세요.
  1. 로그인 사용자만 접근 가능
  2. 특정 역할(admin 등)만 접근 가능
  3. 별도 권한 제어 없음
```

# 확인 후 진행 원칙

- 확인 질문은 **한 번에 모아서** 한다. 항목마다 개별로 질문하여 개발자를 불필요하게 많이 기다리게 하지 않는다.
- 확인 답변을 받으면 그 내용을 기반으로 진행하고, 동일한 내용을 다시 묻지 않는다.
- 확인이 필요 없는 항목은 지체 없이 생성한다. 불필요하게 확인을 요청하지 않는다.
